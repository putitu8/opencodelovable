# ADR 0003: 流式逐步生成管线

## 状态

已接受 (2026-06-16)

## 背景

AI 生成完整应用代码有两种主流架构：

1. **单次批量生成**：一次 API 调用输出所有文件，响应快但复杂应用容易遗漏文件或产生不一致。
2. **流式逐步生成**：AI 先输出计划，再逐个生成文件，用户实时看到构建过程。质量更高但调用链更长。

对于中文版 Lovable 的交互模式（单次 prompt 生成 + 多轮迭代修改），需要平衡生成质量和用户体验。

## 决策

**采用流式逐步生成管线。**

- AI 首先流式输出项目规划（组件树、路由、数据流）。
- 然后逐步生成每个文件（组件、页面、样式、配置），前端实时渲染进度。
- 用户可在生成过程中看到文件逐个出现，体验类似"AI 在手写代码"。
- 多轮迭代时，AI 持有完整项目上下文（已有代码 + 修改指令），同样逐步输出变更。

## 理由

1. **用户体验**：逐步生成让等待变得可见，"看到代码一行行写出来"的体验比 30 秒空白后突然出现的完整应用更好。
2. **生成质量**：AI 可以先规划再执行，减少前后不一致和遗漏。
3. **与 DeepSeek 兼容**：DeepSeek 支持 SSE 流式输出，技术基础具备。
4. **迭代友好**：在迭代场景下，AI 可以精确指出哪些文件被修改、哪些新增，用户有清晰的变更感知。

## 后果

- **正向**：更高的生成质量和更好的用户体验。
- **负向**：API 调用次数增加（规划 + N 个文件 = N+1 次调用），成本更高；生成总时长可能长于单次批量。
- **需关注**：需要实现健壮的流式断线重连机制；对于简单应用（如纯 HTML 落地页），可能需要简化为单次生成以控制成本和延迟。
