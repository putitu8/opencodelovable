# ADR 0002: 统一 Web-first 部署架构

## 状态

已接受 (2026-06-16)

## 背景

最初考虑支持多平台输出：Web（React/Vue/HTML）、微信小程序、React Native/Flutter 移动端。但小程序和原生移动端需要完全不同的构建工具链和部署通道（微信开发者工具 + 小程序 CI、应用商店分发），无法与 CloudBase Webify 的 "Git push → 自动构建 → 一键上线" 模型统一。

可选路径：

1. **真多平台**：为每种输出目标实现独立的构建和部署管线，复杂度极高。
2. **统一 Web-first**：所有输出均为 Web 应用，部署到 CloudBase。移动端和小程序场景生成响应式 H5 或 PWA，不走原生。

## 决策

**所有 AI 生成的应用输出均为 Web 应用，统一通过 CloudBase Webify 部署。**

- 不支持生成原生微信小程序代码（WXML/WXSS）或 React Native/Flutter 原生代码。
- 移动端场景生成响应式 Web/PWA。
- 所有应用共享同一套 CloudBase 部署管线：Git push → Webify 构建 → 上线。

## 理由

1. **Solo 可行性**：维护一套构建部署管线而非三四套，是 Solo 开发的现实选择。
2. **用户体验一致性**：所有应用统一 URL 访问，无需用户在不同的开发者工具和应用商店之间切换。
3. **迭代效率**：修改 → 预览 → 部署的反馈环路上所有应用类型保持一致。
4. **成本统一**：不需要为小程序或移动端额外付费（如小程序服务器费用、App Store/TestFlight 成本）。

## 后果

- **正向**：架构简单，部署管线统一，运维成本低。
- **负向**：无法覆盖需要原生小程序能力的用户（如需要微信支付、获取微信用户信息的场景）；对于强移动端体验的场景，PWA 无法完全替代原生 App。
- **需关注**：当市场反馈强烈需求时，可考虑引入小程序 H5 容器方案（在微信内打开 Web 应用），而非生成原生小程序代码。
